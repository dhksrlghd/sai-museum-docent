# 07. 개발 일지 (Development Journey)

v1.0 → v2.3 까지의 변천을 라운드별로 회고합니다. 각 라운드의 동기·구현·문제·해결을 한 묶음으로.

---

## v1.0 — 단순 RAG 챗봇 데모

### 동기
"한국어로 잘 동작하는 RAG 챗봇을 만들어보자."

### 만든 것
- 큐레이터 추천 321점 스크래핑 (`scrape_one`/`scrape_list`/`scrape_all`)
- 청킹 + e5-small 임베딩 + Chroma (`build_index`)
- 3-mode 시스템 프롬프트 (성인/어린이/외국인)
- React/Vite + Tailwind 단일 챗 페이지
- HF Space Docker 배포

### 문제와 해결
| 문제 | 해결 |
|---|---|
| `from rag import` 모듈을 uvicorn이 못 찾음 | `--app-dir src`로 sys.path 주입 |
| pip install이 4GB CUDA 휠 다운로드 시작 | `pip install torch --index-url https://download.pytorch.org/whl/cpu` 먼저 |
| OpenAI key 짤려서 401 | 토큰 길이 60자 → 정상 160자로 재발급 |
| SSE 토큰이 화면에 안 그려짐 | 파서 구분자 `\n\n` → `/\r\n\r\n|\n\n|\r\r/` |
| Chroma DB 22MB가 git에 안 들어감 | Dockerfile에서 빌드 시 `build_index.py` 재실행 |

### 회고
첫 라운드의 한 줄: **"동작하는 거 → 동작하는 게 데이터 무게가 가벼움"**.
챗 한 화면짜리 데모는 만들어졌지만 사용자(나) 자신이 시연 임팩트가 약하다고 느낌.
"K-Curator는 그냥 API 딸깍 + 소개 끝" 인 느낌.

---

## v2.0 — 박물관 컴패니언 정체성 + 데이터 확장

### 동기
"데이터가 부족해서 정체성이 약하다. 박물관 사이트 다른 영역도 활용하자."

### 만든 것
- **상설전시 스크래퍼** — 7관 36실 / 643점 작품 (`scrape_permanent`)
- **특별전 스크래퍼** — 진행중 5건 (`scrape_special`)
- 통합 청킹: 추천 + 상설 + 특별 → 1,171 → 1,223 청크
- 새 페이지 `/exhibitions` — 진행중 카드 + 층별 트리
- 멀티페이지 라우터 (`/`, `/browse`, `/work/:id`, `/ask`, `/exhibitions`)
- 정체성 변경: "RAG 챗봇" → "박물관 컴패니언"

### 문제와 해결
| 문제 | 해결 |
|---|---|
| 특별전 카드 썸네일이 모두 돋보기 placeholder | hover 버튼 아이콘이 첫 `<img>`로 잡힘 → `btn_more_report` 필터 |
| 작품 상세 우측 AskBox가 모바일에서 깨짐 | `lg:` 접두사로 데스크톱만 sticky |

### 회고
v2.0 끝나고 라이브 배포 후 사용자가 다시 한 번 진단:
> "아쉽단 말이야 도메인이 그냥 API 딸깍 소개 끝 이거잖아…"

데이터는 늘렸지만 **사용 시나리오 차원의 차별화**가 더 필요했음.

## v2.1 — 관람 코스 빌더 (`/plan`)

### 동기
"코스 추천은 다른 박물관 사이트에 없는 기능. 진짜 컴패니언 정체성 굳힘."

### 만든 것
- POST `/api/plan` — SSE 스트리밍 코스 빌더
- 입력: 시간/동반자/관심사
- LLM 시스템 프롬프트: "후보 작품 18점 중에서만 골라라 + 동선 최적화 + 동반자 톤"
- 마크다운 ## H2 / ### H3 / 화살표 / 볼드 출력 → 프론트가 간단 마크다운-라이크 렌더러로 표시
- 헤더 5탭으로 확장

### 회고
"코스 짜기"가 사이트 정체성을 한 단계 끌어올림.
면접에서 한 줄로 설명 가능: *"LLM-agentic course planning grounded in retrieved candidates."*

## v2.2 — 멀티모달 (CLIP) + 매일 큐레이션

### 동기
- 1,423장 이미지가 표시만 되고 검색 자원이 아니었음
- 정적 추천은 재방문 명분 없음

### 만든 것
- `embed_images.py` — 작품당 최대 3장 CLIP 임베딩 (894 시도 / 733 성공)
- 별도 Chroma 컬렉션 `kcurator_images` (cosine, 512-dim)
- API `/api/works/{id}/similar` — 첫 이미지 임베딩으로 닮은 작품 N점
- 작품 상세 하단 "이 작품과 닮은 작품" 6점 그리드
- `daily_pick.py` — 30개 테마 풀 + day-of-year 결정적 회전
- API `/api/today` — 당일 캐시
- 홈 메인 섹션 = 매일 다른 큐레이션

### 문제와 해결
| 문제 | 해결 |
|---|---|
| `clip-multilingual-v1`이 이미지 인코딩 거부 | 표준 `clip-ViT-B-32`로 변경 |
| 임베딩 출력이 버퍼링되어 진행 상황 안 보임 | `python -u` + `flush=True` + `ThreadPoolExecutor(8)` 병렬 다운로드 |
| 박물관 이미지 일부 hot-link 차단 (18% 실패) | 합법적 우회 없이 그대로 진행 (733장도 충분) |

### 회고
멀티모달이 들어가니 "텍스트 RAG 데모"에서 "**모달리티 두 개를 섞은 진짜 멀티모달**"로 이름값.
작품 상세에서 `〈기영회도〉` 보면 다른 계회도가 91% 유사도로 클러스터링되는 모습이 시연 임팩트 큼.

## v2.3 — 마무리 (Phase D)

### 동기
"기능은 풍부한데 데이터에 빈틈이 있고, 모바일이 깨져서 신뢰가 부족."

### 만든 것
- **상설실 소개 보강** — 한국어 비율 필터 적용한 새 파서 → 2,994자 → **23,314자** (8배)
- **추천↔상설 매칭** — 52점 추천 작품에 hall/floor/room_name 메타 부착
- 모바일 햄버거 메뉴 (md 미만)
- 작품 상세 우측 AskBox 모바일 stack
- README 전면 개정 (시나리오 표, 데이터 풍경, 25분 빌드 가이드)
- 청크 1,223 → 1,265

### 회고
"검색 품질 측정 가능한 개선" — `사유의 방` 쿼리가 v2.2에선 무관한 작품 잡았는데 v2.3에선 **사유의 방** 청크 직접 매칭 (cosine 0.876).

## v2.4 — 리브랜딩: K-Curator → 사이 (SAI)

### 동기
"K-Curator는 정직한 기능 설명이지만 마음을 못 흔든다. 한국적 정서 + 시적 임팩트."

### 만든 것
- 6개 후보 (결, 곁, 사유첩, 사이, 온고, 한뜸) 중 **사이** 선택
- 헤더 로고 `사  이` (letter-spacing 0.5em — 글자 사이 빈 공간이 곧 컨셉)
- 부제 "작품과 당신 사이"
- 홈 히어로: "작품과 당신, / 그 사이를 잇다"
- LLM 시스템 프롬프트의 자기소개 변경 ("I'm SAI, meaning in-between")
- 모든 한국어 텍스트에 `break-keep` (word-break: keep-all) 적용 — 단어 중간 끊김 방지

### 회고
이름이 바뀌면 정체성이 바뀜. "사이"는 RAG·멀티모달·코스 빌더 모두 "사이를 잇는" 행위로 통일적으로 설명됨. 이름이 컨셉 그 자체.

## 무엇을 배웠나

1. **데이터 품질 > 데이터 양**. 33만 e뮤지엄 노이즈 < 321 큐레이터 본문.
2. **챗봇은 출발이지 도착이 아니다**. 페이지 분리·시나리오 분리가 정체성을 만든다.
3. **모달리티는 자산이다**. 1,423장 이미지를 표시만 한다면 절반의 RAG. CLIP 도입이 차별화 카드.
4. **결정적 단순함이 cron보다 낫다**. 매일 픽은 day-of-year 인덱싱 한 줄로 충분.
5. **이름이 정체성이다**. K-Curator → 사이 한 번의 리브랜딩이 사이트 무게감을 바꿈.
6. **무료 인프라의 함정**: CUDA 휠, hot-link 차단, 디스크 한도 — 모두 한 번씩 부딪침.
7. **SSE는 관대하게 파싱**. `\r\n\r\n` 사건이 큰 교훈.
